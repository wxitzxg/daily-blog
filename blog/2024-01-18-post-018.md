如果你之前在 Claude Code 里已经积累了一堆技能（skills）、流程（workflow）、脚本（scripts）、提示词模板（prompts），那你可能体验过一种“明明很强但总差一步”的尴尬：  
它们都要你亲自去点开、输入、触发、盯着跑。

Scheduled Tasks 的出现，补上的就是这一步： **让你创建的能力自动运行。**

> "Anthropic finally did it: native Scheduled Tasks."  
> "Anthropic 终于做到了：原生推出 Scheduled Tasks（定时任务）。"  
> — 00:32（视频时间）

为什么是“10 倍”？因为从这一刻开始，你的工作流不再依赖你“在线”。你只需要设计一次，剩下就是自动触发、自动跑完、自动结束。

这也是很多人第一次意识到：所谓“AI 自动化”，真正的分水岭不是“能不能做”，而是—— **能不能在你不在的时候做** 。

## 02 定时任务到底怎么创建？比你想的简单，但比你想的更危险

![配图](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

Scheduled Tasks 目前有一个硬前提： **必须在 Claude Code 桌面端（Desktop App）里用** 。  
（后面会讲为什么这很关键，以及它带来的第一个大坑。）

创建方式有两种：

1）在顶部找到 **Schedule** 标签页，点 **New task** 新建任务  
2）在任意会话里直接输入命令：\`/schedule\`

界面里你要填的东西基本就是“让它像员工一样上班”需要的要素：

\- **name** ：任务名字（比如“morning coffee”）  
\- **description** ：任务描述（告诉未来的你它是干嘛的）  
\- **prompt** ：核心提示词（它到底要做什么）  
\- **model** ：选择运行模型  
\- **mode** ：运行模式（更偏 agent 还是偏执行）  
\- **folder** ：它可访问/工作的项目目录

最后一步是关键：选择时间规则。你可以按 UI 选“每小时/每天/每周/指定时间”，背后实际就是 cron 规则。

一旦时间到了，会发生什么？

\- 系统按 cron 触发  
\- **启动一个全新的 session（新会话）**  
\- Claude 读取你的 prompt  
\- 遍历你允许的文件、在项目里工作  
\- 完成后自动停止并结束

这段话看似平静，但它背后意味着：你不再是“点按钮的人”，你变成了“制定制度的人”。从执行者变成设计者——这就是生产力跃迁的开始。

## 03 最大解锁点：它不是“跑脚本”，它是“会自愈的智能体工作流”

![配图](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

作者说他最兴奋的地方只有一句话： **这不是确定性工作流（deterministic workflow）。**

> "The huge unlock for me is: this is not a deterministic workflow."  
> "对我来说最大的解锁点是：这不是一个确定性的工作流。"  
> — 03:12（视频时间）

什么意思？你以前做自动化，大概率是这样的：

\- 写一个 Python/TypeScript 脚本  
\- 设定：第 1 步 → 第 2 步 → 第 3 步  
\- 一旦报错：停止、抛错、通知你  
\- 你来修

这叫确定性：它执行的是代码，不会“动脑子”。

但 Scheduled Tasks 跑的不是“脚本”，而是一个像你平时聊天那样工作的 Claude Code 智能体。它会读你的项目全局，会用工具，会试错，会换方案。

> "It's a Claude Code agent running the same way you chat with it."  
> "它运行的是 Claude Code 智能体，方式和你平时与它对话时一样。"  
> — 03:40（视频时间）

更重要的是： **它可以自我修复（self-healing）** 。  
这句话听起来像营销，但作者举了一个特别具体的行为模式：它遇到错误不会“认输”，它会改策略。

> "If it hits an error, it won't just come back and say it tried."  
> "如果它遇到错误，它不会只回来对你说‘我试过了’。"  
> — 04:22（视频时间）

它会做的是：再试三种方式，找到最稳的那种，然后把经验“写回”到流程里——保证下次不再踩坑。

当这种自动化开始“自愈”和“变好”，真正改变的是你的角色：你不再是瓶颈。你不需要每天守着它运行、等它报错、手动修复。

当然，如果你特别追求可控性，也不是没办法：你可以让定时任务只做一件事—— **只运行一个脚本** 。这样就把它拉回确定性，像传统自动化一样可预测。

但你要清楚：一旦你体验过智能体式的“会修会学”，你很难再回到只会按步骤执行的自动化。

## 04 “morning coffee”案例：从每天手动输入，到每天 6 点自动开工

![配图](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

作者展示了一个很典型、也很“像真实工作”的任务：morning coffee。

以前他的早晨是这样开始的：

\- 打开 Claude Code  
\- 输入“运行 morning coffee”  
\- Claude 帮他规划一天、看承诺安排、扫项目进度、跟上团队动态

现在它可以变成： **每天早上 6:00 自动运行** 。  
他做的配置甚至朴素到离谱：直接对 Claude 说——“看一下我的 morning coffee 技能，我想把它变成每天早上 6 点触发的定时任务，你帮我设置一下。”

Claude 读完技能，问一个确认问题，然后就配好了。

你可以想象这种体验：你不是在“做自动化”，你是在“教会一个助理怎么每天早上给你递咖啡和日程表”。

更夸张的是，如果你把项目导入 Claude Code（GitHub 仓库或本地文件夹都可以），你甚至可以像问同事一样问它：

\- 这个项目里有哪些 skills？  
\- 这一类技能能不能每周跑一次？  
\- 把某个 research 流程改成每晚 11 点执行？

这时候你会发现：所谓“Claude Code 2.0”的感觉并不是模型更聪明，而是—— **你终于有了一个能定时上班的 AI 角色** 。

## 05 你必须提前知道的 3 个大坑：不开机、要权限、还无状态

![配图](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

热血归热血，但 Scheduled Tasks 目前有几个限制，不知道的话很容易翻车。

### 1）电脑必须开机，桌面端必须打开

这是第一个大坑：你关机，它就不跑。你关掉桌面端，它也不跑。

但有一个“补救机制”：如果你错过了任务，Claude Code 会回看过去 7 天的错过任务，然后补跑。  
这对“日报总结、周报汇总”这种不那么强时效的任务很友好；但对“准点抢票/准点发布”这种就不完美。

### 2）它无人监督运行：权限不管好，分分钟跑偏

定时任务的理想状态是：不需要它中途问你问题。 可现实是：一旦它需要权限、需要你确认、或者碰到敏感操作，它可能会暂停。

所以你要做的是两件事：

\- **用权限（permissions）把它关进笼子** ：比如禁止删除文件、禁止危险 bash 命令、禁止对仓库做大范围不可逆改动  
\- 让它知道边界：它能做什么、不能做什么、遇到什么必须停下来

作者提到你可以通过修改项目本地 settings 来约束它，甚至可以设置规则：拒绝任何包含 delete/remove 的命令。

### 3）每次运行都是新会话：无状态（stateless）

这点非常关键：Scheduled Task 每次触发都会启动一个全新 session，不带上一次的上下文。

好处是：干净、稳定、不容易被旧信息污染。  
坏处是：你想让它“越跑越懂你”，就必须自己设计“记忆机制”。

而且还有一个非常现实的坑：如果你没提供 API key，或者任务某步需要额外授权，它就会停下来等你——自动化直接失去意义。

所以作者给了一个非常硬核、非常实用的建议： **新任务创建完，立刻手动跑一遍** 。

> "Once you create a new task, run it manually once to make sure it completes without you."  
> "一旦创建了新任务，立刻手动运行一次，确保它能在无需你介入的情况下跑完整个流程。"  
> — 09:18（视频时间）

这一步做完，你才知道它会不会卡在权限、卡在 API、卡在“需要你选择”的岔路口。

## 06 想让它越跑越强？用“日志文件”给它造记忆，再用 Hook 让你听见它下班

![配图](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

Scheduled Tasks 的终极想象不是“定时执行”，而是“定时进化”。

作者提了一个非常清晰的自我改进闭环，我把它翻译成更可操作的三层结构：

### 第一层：修复脚本本身（代码自愈）

如果任务里包含脚本，那么你可以在 prompt 中明确要求： 遇到错误时，先定位原因，再编辑代码修复，再重新运行验证。

这一步会把“报错→你修”变成“报错→它修”。

### 第二层：优化 prompt（策略自愈）

很多自动化失败不是因为代码错，而是因为 prompt 写得不够稳定。 你可以让它在每次任务结束时复盘：哪里指令模糊？哪里约束不够？哪里容易误解？然后主动改写 prompt。

这听起来很玄，但其实就是：让它像一个不断优化 SOP 的运营。

### 第三层：用日志当记忆（解决 stateless）

因为每次都是新 session，它“天生失忆”。那怎么办？用文件。

你可以选择两种日志策略：

\- **追加式 log（append）** ：每次把运行结果追加到一个日志里  
\- **覆盖式 log（overwrite）** ：每次用最新一份“上次运行摘要”覆盖旧内容

作者问 Claude Code 该怎么做，Claude 给出的建议是“精益策略”：每个任务对应一个文件，每次运行都覆盖写入，记录上次运行情况、耗时、做了什么、已知问题、下一步建议等。避免跑 1000 次生成 1000 条日志把自己淹没。

更关键的是 prompt 结构要固定成三段：

1）开始前：先读取“上次运行文件”  
2）执行中：完成主要任务  
3）结束后：把本次结果覆盖写回“上次运行文件”

这就是你给定时任务装上的“记忆器”。

---

### 以及一个特别现实的细节：你怎么知道它跑完了？

桌面端确实会有通知，但作者吐槽体验一般：不够“打断你”，没声音，容易错过。

所以他做了两件事，非常值得抄作业：

1）加 Hook：每次 Claude Code session 结束就播放系统提示音  
2）在技能 prompt 末尾加一句：完成后给我发一条 ClickUp 消息（或你常用的工具）告诉我已完成

这样你就能真正做到：AI 在后台跑，你在前台干别的；但它一结束，你能立刻收到可感知的反馈。

## 结尾：从今天起，你不缺工具，你缺一个“会自己上班”的系统

Scheduled Tasks 真正让人上头的不是“定时”，而是它把 Claude Code 从“随叫随到的助手”，推成了“按制度运转的员工”。

你可以从最简单的任务开始：

\- 每天早上自动生成今日计划 + 项目风险提醒  
\- 每天下班自动汇总 commit / PR / issue，写一份日报  
\- 每周自动扫一遍代码库的 TODO，输出清单并按优先级排序  
\- 每晚自动跑一次测试/脚本，失败就给你发消息并附带修复建议

但我也想把最重要的一句提醒再强调一次： **先手动试跑一次** 。别让自动化在凌晨卡在权限弹窗里，然后你第二天早上以为它已经做完了。
