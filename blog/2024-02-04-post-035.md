原创 Lex 陆徐洲 *2026年3月31日 08:15*

OpenAI 居然给自己的竞争对手做了一个插件。

OpenAI 在 3 月 25 号上线了 Codex 插件系统，紧接着 3 月 30 号就发布了一个官方插件叫 codex-plugin-cc，cc 就是 Claude Code 的缩写。装上之后，你可以在 Claude Code 里直接调用 Codex 来审查代码、委派任务。

两家打得最凶的 AI 编程工具，现在可以在同一个终端里协作了。

大家好，我是陆徐洲。看到这个消息的时候，我第一反应是"这也行？"，第二反应是"赶紧装上试试"。

说说这个插件能干什么。核心就三个命令：

• `/codex:review` ：让 Codex 审查你当前的代码变更，相当于找了个 OpenAI 的工程师帮你 code review• `/codex:adversarial-review` ：对抗性审查，不只是找 bug，而是质疑你的设计决策• `/codex:rescue` ：把任务直接丢给 Codex 后台执行，比如"帮我修掉所有 SQL 注入"

简单一句话，就是让 Claude 写代码，让 Codex 来挑刺。两个 AI 互相审查，比单个 AI 自检靠谱。

安装过程看起来很简单，4 条命令：

```bash
/plugin marketplace add openai/codex-plugin-cc/plugin install codex@openai-codex/reload-plugins/codex:setup
```

Mac 用户到这里应该就结束了。

我不是 Mac 用户。

跑完 `/codex:setup` ，状态表四行全红。Node.js 有，npm "Not found"，Codex CLI "Not installed"，Auth "Not authenticated"。

![图片](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

离谱的是，我明明装了 npm，也装了 Codex CLI，在 PowerShell 里跑 `npm --version` 和 `codex --version` 都正常。就是这个插件检测不到。

卡就卡在一个 Windows 的经典问题上： `npm` 和 `codex` 在 Windows 上不是 `.exe` 文件，而是 `.cmd` 批处理脚本。插件用 Node.js 的 `spawnSync("npm")` 去找它们，不加 `shell: true` 参数的话，Windows 上就是找不到 `.cmd` 文件。

这是 Node.js 在 Windows 上的老坑了，做过跨平台开发的应该都踩过。

修复方法很暴力，直接改插件源码，给 `spawnSync` 加一个 `shell: process.platform === "win32"` 。一行代码的事。

改完再跑 `/codex:setup` ，终于可以正常配置了。

第二个坑来了。跑 `/codex:review` 的时候，任务卡在 "starting" 状态不动，等了 15 分钟没反应。排查发现 `app-server.mjs` 里也有一个 `spawn("codex")` ，同样的问题。

再改一处，改完再跑又遇到第三个坑。兜兜转转折腾了将近两个小时，最终在 Windows 11 上没有成功跑通完整的 review 流程，感觉后续应该到github上提个issue。

如果是 Mac 或 Linux，大概 5 分钟就搞定了。

但比起踩坑本身，我更想聊聊这件事背后的信号。

**OpenAI 主动给 Claude Code 做插件，这在半年前是不可想象的。** Codex 和 Claude Code 是目前 AI 编程领域打得最凶的两家，各有各的生态和用户群。现在 OpenAI 不但没有封闭生态，反而把自己的能力包装成插件，送到对手的地盘上去。

为什么？因为 Claude Code 的用户基数摆在那里。日活 GitHub commit 13.5 万次，年化收入 25 亿美金。Codex 如果只守着自己的客户端，就是放弃了一个现成的巨大市场。与其让用户二选一，不如让 Codex 的强项（代码审查、对抗性测试）直接嵌入到用户已有的工作流里。

这个思路其实不新鲜。很多人已经在 Claude Code 里通过 Skill 调用 Codex CLI 做代码审查了，codex-plugin-cc 只是把这个野路子变成了官方支持。

更值得关注的是，这可能只是开始。

Codex 插件系统上线不到一周，已经有 Gmail、Slack、Figma、Notion、Google Drive 等十几个官方插件了。Claude Code 这边也有自己的插件市场和 MCP 生态。两套系统都在往"平台化"的方向走，而 codex-plugin-cc 是第一个跨阵营的桥梁。

顺着这个逻辑想， **Gemini CLI 也该跟进了** 。Google 的 Gemini 3.1 Pro 在设计能力上其实很能打，但 Gemini CLI 的生态还很薄。如果 OpenAI 都能放下身段给竞品做插件，Google 没理由不做。哪家先把自己的强项做成可插拔的模块，送到开发者已有的工作流里，哪家就赢了下一轮。

AI 编程工具的竞争，正在从"谁的模型更强"变成"谁的生态更开放"。

这不是第一次了。Claude Code 早期版本对 Windows 支持也不好，Codex CLI 也是先有 Mac 版再补的 Windows。整个 AI 开发工具生态，Windows 永远是"也支持"的那个。不是 Mac 更好，是这些工具先给 Mac 做的。

用 Windows 做 AI 开发，就是在帮开源社区做 beta 测试。也许真的该换 Mac 了。

我是陆徐洲，一家 LIMS 公司的 AI 算法负责人。关注我，让我们一起在 AI 落地实践的路上，走得更远。
