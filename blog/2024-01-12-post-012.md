## AI编程的三个"坑"

用过AI编程的朋友，大概都踩过这几个坑：

**坑一：需求在聊天里丢了**

AI会记住你说过的话，但聊着聊着上下文就乱了。你说"先做个用户登录"，过20分钟说"再加点权限管理"，AI可能已经把两件事混在一起做了，因为它记不清你到底想要什么。

**坑二：边做边猜**

你让AI"加个暗黑模式"，它可能同时干了：改CSS变量、加toggle按钮、改状态管理、调整主题配置……一通操作猛如虎，结果十个功能里只有三个是你要的。

**坑三：改着改着就回不去了**

代码越改越多，想回退但不知道改了什么。只能硬着头皮继续改，或者干脆删掉重写。

这些问题不是因为AI不够聪明——恰恰相反，是 **AI太聪明了，聪明到忍不住自己发挥。**

## 规范驱动编程：让AI先"想清楚"再动手

这时候就需要 **规范驱动编程** （Spec-Driven Development）了。

听起来高大上，其实道理很简单： **先想清楚做什么，写成文档，再让AI动手。**

就像装修房子：你要先出设计图、列好材料清单、施工图，然后工人再干活。而不是跟工人说"帮我装一下"，然后祈祷他装出你想要的样子。

这就是OpenSpec在做的事。

---

## OpenSpec是什么

OpenSpec是一个开源的命令行工具，专门给AI编程加了一个"规范层"。

它不替代你的AI编程工具——你继续用Cursor、Claude Code、Copilot都行。OpenSpec的作用是 **让AI先跟你对齐想法，再开始写代码。**

它的核心工作流只有四步：

```
1. Proposal（提案）→ 为什么要做这个功能
2. Spec（规格）     → 具体要做成什么样
3. Apply（实施）   → 按规范生成代码
4. Archive（归档） → 合并回主规范
```

---

## 举个例子：怎么用OpenSpec

假设你想在项目里加一个"用户搜索过滤"功能。

**第一步：发起提案**

你跟AI说：

> 创建一个OpenSpec变更提案，用于添加用户搜索过滤器

AI会自动创建一个规范目录：

```
openspec/
 └── changes/
      └── add-user-search-filter/
           ├── proposal.md      # 为什么做
           ├── tasks.md         # 怎么做
           └── specs/
                └── user/
                     └── spec.md   # 具体规格
```

**第二步：审核规范**

你说：

> 为搜索过滤器添加验收标准：支持按姓名搜索、按部门筛选

AI会更新规范文档，然后你俩反复改，直到都认可这个方案。这个过程叫"对齐"—— **在写代码之前，先确认大家理解一致。**

**第三步：执行**

确认规范没问题后，运行：

```
/openspec:apply add-user-search-filter
```

AI开始按规范生成代码，一次只做一个任务，不会顺便"发挥"。

**第四步：归档**

功能测试通过后，运行：

```
/openspec:archive add-user-search-filter
```

所有变更合并回主规范，保留完整历史记录。

| 以前 | 用OpenSpec之后 |
| --- | --- |

说白了， **OpenSpec不让AI变得更聪明，只让它更听话。**

## 哪些AI编程工具可以用OpenSpec

这也是很多人关心的问题：OpenSpec支持哪些工具？我常用的能直接用吗？

好消息是： **主流的AI编程工具它都支持。**

| 工具 | 支持情况 | 特点 |
| --- | --- | --- |
| **Cursor** | ✅ 原生支持 | 自动注入OpenSpec命令 |
| **Claude Code** | ✅ 原生支持 | 可用 `/openspec:proposal` 命令 |
| **GitHub Copilot** | ✅ 原生支持 | 通过命令行集成 |
| **OpenCode** | ✅ 支持 | 开源免费CLI工具，可连接任意LLM |
| **VS Code + AI插件** | ✅ 支持 | 通用方式调用 |
| **Zed** | ✅ 支持 | 通过命令面板 |
| **Aider** | ✅ 支持 | 命令行AI编程工具 |

> OpenCode是开源的CLI编程工具（opencode.ai），完全免费，支持连接任意LLM（OpenAI、Claude、Ollama等），完全支持OpenSpec。如果你不想用Cursor或Copilot，OpenCode是个不错的选择。

**简单说：** 只要你用的是主流AI编程工具，基本都能无缝接入OpenSpec。

## 技术干货：OpenSpec是怎么工作的

如果说前面都是"用法"，这部分聊聊"原理"——看看OpenSpec到底是怎么让AI听话的。

### 1\. 规范注入（Spec Injection）

当你运行 `openspec init` 时，它会：

- 在项目根目录创建 `openspec/` 目录
- 生成一个`.openspec.json` 配置文件
- 自动配置你的AI工具，注入系统提示词

这个配置文件里包含了：

- 项目规范的位置
- 当前处于哪个变更阶段
- 任务清单模板
- 验收标准格式

### 2\. 变更追踪（Change Tracking）

每一次你发起一个提案，OpenSpec都会创建一个独立的变更目录：

```
openspec/
├── specs/                    # 主规范（真相来源）
│   ├── README.md
│   └── system/
├── changes/                  # 所有变更历史
│   ├── add-user-search/
│   ├── fix-login-bug/
│   └── update-api/
└── .openspec.json           # 配置文件
```

这就像Git一样—— **但不是追踪代码变更，而是追踪需求变更。**

### 3\. 任务拆分（Task Breakdown）

当你执行 `/openspec:apply` 时，OpenSpec会：

1. 读取 `tasks.md` 中的任务列表
2. 一次只给AI分配一个任务
3. 完成后才进入下一个
4. 记录每个任务的完成状态

这样AI就不会"一锅炖"，而是按部就班。

### 4\. 验收标准（Acceptance Criteria）

规范文档里最关键的部分是验收标准，格式大概是这样：

```
## 验收标准

### 功能验收
- [ ] 用户可以通过姓名搜索
- [ ] 用户可以按部门筛选
- [ ] 搜索结果实时更新

### 性能验收
- [ ] 搜索响应时间 < 200ms
- [ ] 支持1000+用户同时搜索
```

AI在实现时会对标这些标准，完成后可以自检。

## 怎么开始

很简单，两分钟就能用：

```
# 安装
npm install -g @fission-ai/openspec@latest

# 初始化
cd your-project
openspec init
```

然后就可以用了。如果你用Cursor、Claude Code或Copilot，它会自动添加 `/openspec:proposal` 这样的命令。

没有也不影响，直接说"创建一个OpenSpec提案"就行。

---

## 什么人适合用

- **经常用AI编程**
	但受够了它"自说自话"
- **项目有一定复杂度**
	不是几个文件那种
- **想规范化团队AI使用流程**
	的

如果你的项目只是个简单脚本，几句话就能说清楚需求，那可能没必要用。OpenSpec适合 **需要认真规划的功能开发。**

## 总结

AI编程确实快，但"快"和"对"是两码事。

OpenSpec解决的问题很简单： **不要让AI猜你想要什么，先写下来，让它照着做。**

就像那句话说的：AI不会取代程序员，但会用规范的程序员会取代不会用的。

技术本身不复杂，两分钟就能上手。感兴趣的可以试试，看看AI是不是真的变听话了。
