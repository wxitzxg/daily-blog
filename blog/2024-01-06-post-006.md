## 一个问题：为什么 AI 编程总是不靠谱？

你有没有这种体验：

让 Claude Code 帮你写个项目，刚开始挺好的，写着写着就开始 **犯糊涂**  

```javascript
忘记之前的约定重复犯同样的错误输出越来越"简洁"（其实是偷工减料）需要反复提醒它注意什么
```

**这叫 Context Rot（上下文腐烂）。**

简单说：当对话太长，上下文窗口被填满时，AI 输出质量会断崖式下跌。

**Get Shit Done（GSD）就是来解决这个问题的。**

## 02

## GSD 是什么？

**一句话：让 Claude Code 真正能成事的元提示系统。**

![图片](data:image/svg+xml,%3C%3Fxml version='1.0' encoding='UTF-8'%3F%3E%3Csvg width='1px' height='1px' viewBox='0 0 1 1' version='1.1' xmlns='http://www.w3.org/2000/svg' xmlns:xlink='http://www.w3.org/1999/xlink'%3E%3Ctitle%3E%3C/title%3E%3Cg stroke='none' stroke-width='1' fill='none' fill-rule='evenodd' fill-opacity='0'%3E%3Cg transform='translate(-249.000000, -126.000000)' fill='%23FFFFFF'%3E%3Crect x='249' y='126' width='1' height='1'%3E%3C/rect%3E%3C/g%3E%3C/g%3E%3C/svg%3E)

**GitHub：**

```apache
35,000⭐ 单日增速：+2,642⭐
```

**用户评价：“Literally just gets shit done"**

**核心思路很简单：**

把大任务拆成多个小计划， **每个计划都在全新的上下文窗口中执行** 。

用作者的话说：

> "Each plan is small enough to execute in a fresh context window. No degradation."

**效果：** 每个任务的执行质量一致，不会因为上下文积累而下降。

## 03

## 怎么用？4 个命令搞定

### 第一步：

```javascript
/gsd:new-project
```

**一个命令，从想法到路线图。**

系统会追问你：

```js
目标是什么？有什么约束？技术偏好？边界情况？
```

然后生成：

```js
PROJECT.md（项目描述）REQUIREMENTS.md（需求文档）ROADMAP.md（分阶段路线图）
```

**我的体验：** 提问环节很细，但比直接说"帮我做个XX"靠谱太多。

---

### 第二步：

```bash
/gsd:discuss-phase 1
```

**这是** **GSD** **最被低估的功能。**

路线图里每个阶段只有一两句话，不够。这个步骤帮你 **把模糊想法变清晰** 。

系统会根据你在构建的内容，自动识别灰色地带：

```js
视觉功能 → 布局、交互、空状态APIs → 响应格式、错误处理内容系统 → 结构、语气、流程
```

**对每个领域追问到你满意为止。**

**输出：** CONTEXT.md — 直接喂给后续步骤。

**作者原话：**

> "The deeper you go here, the more the system builds what you actually want."

**我的体验：** 这个步骤花时间最多，但最值得。我之前用 Claude Code，经常做到一半发现“这不是我想要的"，GSD 把这个风险前置了。

---

### 第三步：

```bash
/gsd:plan-phase 1
```

系统会：

```js
研究— 调查如何实现规划— 创建原子任务计划验证— 对照需求检查输出：每个任务都是原子级的，执行起来很清晰。
```

---

### 第四步：

```bash
/gsd:execute-phase 1
```

系统会：

```sql
分波执行— 能并行的并行独立上下文— 每个计划 200K tokens 纯用于实现原子提交— 每个任务一个 git commit输出：干净的 git 历史，完成的工作。
```

## 我的实战测试

### 项目 1：CLI 工具（小型）

- 规模：500 行代码
- 时间：40 分钟
- 结果：✅ 一次性成功

**感受：** 对小项目来说提问环节略显繁琐，但代码结构清晰。

---

### 项目 2：Web 应用（中型）

- 规模：3,000 行代码
- 时间：3 小时
- 结果：✅ 分 3 阶段完成

**感受：** 这是 GSD 的甜蜜点。项目足够复杂，需要系统化规划。

---

### 项目 3：重构代码库（大型）

- 规模：10,000 行代码
- 时间：8 小时
- 结果：⚠️ 部分成功，需手动调整

**感受：** 规划能力可靠，但边界情况需要人工介入。

## 07

## 值不值得装？

### 值得装，如果：

- ✅ 你用 Claude Code 频率高（每周 3 次+）
- ✅ 项目有一定复杂度
- ✅ 愿意在前期花时间明确需求
- ✅ 受够了"做到一半发现不是想要的"

### 不值得装，如果：

- ❌ 只是偶尔用用
- ❌ 项目都很简单
- ❌ 偏好"边做边改"
