## BMAD 官方完整流程：我实测下来的真实感受

### 安装和触发

官方标准命令就上面那行。安装后在 Claude Code/Cursor 中打开项目，输入 `/bmad-help` 获取引导。

当时我的需求是：搭建一个专利撰写辅助系统，覆盖专利选题、检索、分段撰写、审计评分全流程。

用 BMAD 开发，我只需要输入：

```
使用 BMAD 开发：专利撰写辅助系统，覆盖专利选题、专利检索、分段撰写、审计评分全流程
```

然后它就带着我走了。

### Spec 生成全流程（一步不差）

**1.** ：定义问题、用户、MVP 范围

BMAD 会自动追问：用户场景、核心痛点、专利类型、目标领域。一次一问，深度挖掘。说实话，这个追问过程让我想清楚了很多一开始没考虑到的细节，比如发明专利和实用新型的差异处理。

**2.** ：生成完整需求文档

含用户画像、验收指标、风险点。它自动标注了专利检索的合规风险：需要对接官方数据库，不能只用公开数据源。这个我当时真没想到。

**3.** ：输出技术架构

提供 3 套方案，明确检索 API 对接、分段撰写模板、评分模型等细节。我最后选了中间那个方案，平衡了准确性和成本。

**4.** ：拆解为可执行用户故事

把专利撰写系统拆分为：选题推荐、检索分析、分段撰写、审计评分、报告生成等。优先级都标好了，直接开干。

**5.** ：迭代跟踪初始化

指定交付标准，后续跟进就有据可依了。

### 开发与审查闭环

- • `/quick-spec` ：分析代码库，生成技术规格与故事
- • `/dev-story` ：按故事开发，子 Agent 分工执行
- • `/code-review` ：内置 Quinn(QA) 自动测试，质量校验
- • 可选 TEA 企业级测试模块：风险驱动测试策略、发布闸门

### 最终产出

完全符合专利撰写全流程需求，选题→检索→撰写→审计闭环，自带测试用例，可直接上线。

### 真实使用感受

被专业团队带着走，从产品→架构→开发→测试全流程标准化。不是简单代码规范，是 AI 时代的敏捷开发完整方法论。大项目/团队协作碾压轻量工具，新手也能产出企业级质量。

说实话，跑完这套流程，我觉得 BMAD 最适合的是那种需要从 0 到 1 完整构建的产品，尤其是团队协作场景。

## GStack 官方完整流程：模式切换太丝滑

### 安装

官方标准命令：

```
git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack && cd ~/.claude/skills/gstack && ./setup
```

需在 CLAUDE.md 中添加 gstack 配置，指定使用 `/browse` 等技能。

### 触发需求

需求同上，先进入规划模式：

```
开发专利撰写辅助系统，覆盖专利选题、检索、分段撰写、审计评分全流程
```

### 6 大 Skill 官方用法

**1\. /plan-ceo-review（创始人模式）**

不直接实现需求，而是重新思考用户价值。它说：专利撰写的核心不是堆字数，而是提高授权率、降低驳回风险。直接优化出选题热度分析、prior art 检索、权利要求分层、审计打分等 10 星体验。

这个视角我当时真没想到，光想着怎么实现功能，忘了重新思考用户价值。

**2\. /plan-eng-review（技术负责人模式）**

输出架构图、状态机、数据流、边界设计、测试矩阵，明确：

- • 专利检索 API 对接方案
- • 分段撰写模板引擎
- • 审计评分模型设计
- • 异常处理与重试逻辑

**3\. /ship（偏执工程师模式）**

开发完成后审查生产级风险：

- • 并发检索冲突问题
- • 数据合规与隐私边界
- • 临时文件清理
- • 专利格式兼容性

**4\. /browse（发布工程师模式）**

自动同步主分支、运行测试、推送、创建 PR，一键发布。这个是真的解放双手。

**5\. （QA 工程师模式）**

自动打开浏览器，登录专利数据库、遍历检索、截图、校验控制台，60 秒完成全量 QA。这个直接封神，我测的时候都惊了。

**6\. /retro（工程管理模式）**

分析提交历史、速率、质量，生成复盘报告，持续优化流程。

### 最终产出

生产级质量，无隐藏 Bug，专利撰写全流程覆盖，自带完整测试与发布记录，适合高频迭代的生产环境。

### 真实使用感受

模式切换太丝滑，需要产品高度就切 CEO 视角，需要技术严谨就切 Eng 视角。审查和发布完全解放双手，浏览器自动化 QA 直接封神，Claude 重度用户必装。

## 6 款工具核心差异（基于官方源码）

跑完这六款，我大概理清楚了它们的定位：

| 工具 | 核心定位 | 官方工作流 | 适配平台 | 最佳场景 |
| --- | --- | --- | --- | --- |
| BMAD | AI 敏捷开发全框架 | product-brief→prd→architecture→dev-story→review | 全平台 AI IDE | 企业级、团队协作、完整产品从 0 到 1 |
| GStack | Claude 专家工作流 | 6 大 Skill：规划/评审/审查/发布/浏览/复盘 | 仅 Claude Code | 高频发布、生产级、需 QA 自动化 |
| Superpowers | 通用工作流系统 | 14Skill：头脑风暴→隔离→拆解→子 Agent→多轮审查 | 全主流 Agent | 个人开发、通用场景、新手友好 |
| SpecKit | GitHub 官方 Spec 驱动 | 结构化工件→一致性校验→实现→核查 | 18+AI 工具 | 跨平台、规约一致性、团队规范 |
| OpenSpec | 存量代码 Spec 驱动 | 一变更一文件夹、迭代式开发、隔离变更 | 20+AI 工具 | 已有项目迭代、多需求并行 |
| everything-claude-code | Claude 全配置生产系统 | 多 Agent 编排、规则架构、内存持久化 | 仅 Claude | Claude 专属、生产级优化 |

## 选型建议（我踩坑后的真实建议）

1. 1\. **完整产品/团队协作/企业级** → BMAD（真正的 AI 敏捷框架，34k Star 不是虚的）
2. 2\. **Claude 专属/生产发布/QA 自动化** → GStack（YC CEO 亲测，发布与审查无敌）
3. 3\. **个人通用/新手/全平台适配** → Superpowers（11 万 Star，无脑冲）
4. 4\. **跨平台/规约统一/GitHub 生态** → SpecKit
5. 5\. **存量代码/多需求迭代** → OpenSpec
6. 6\. **Claude 深度定制/生产优化** → everything-claude-code

说实话，我当时选的时候也纠结了很久，最后发现关键不是哪个工具最厉害，而是哪个最适合你的场景。

## 最后说句实话

Spec 驱动的核心从来不是让 AI 写代码更快，而是先把需求想透、把规范定死、把流程闭环，避免返工。

BMAD 是完整开发体系，GStack 是 Claude 专家增强，Superpowers 是通用工作流天花板，三款完全不是一个维度，按场景选，装对一个，你的 Agent 体验直接质变。
